Methods to Control the Robot Autonomously

There are 3 main methods to control the SO101 robotic arm autonomously:

  1. Motion Replay 🔄 - Record and replay pre-programmed movements
  2. Inverse Kinematics (IK) 🎯 - Calculate joint angles to reach target positions
  3. ACT Model 🧠 - Action Chunking Transformer, a learning-based approach

Motion Replay

Motion replay is the simplest method for controlling the SO101 arm.

How it works:

  • Motion data can be captured via teleoperation (human guides the arm)
  • Or by moving the arm directly and recording joint positions

The recorded movements are then replayed automatically by the robot.

Limitations of Motion Replay

Problem: If the object that needs to be picked up is moved, the replayed motion will fail.

  • Fixed trajectory cannot adapt to changes
  • Requires precise object positioning
  • Any variation causes failure

More Limitations: Combinatorial Explosion

Using a chess robot as an example:

  • 64 squares on a chessboard
  • 32 pieces to potentially move
  • Possible pick-and-place operations: 32 × 64 = 2,048 combinations

This implies thousands of recordings would be needed to cover all scenarios!

  • Recording every possible move is impractical
  • Storage requirements grow exponentially
  • Maintenance becomes a nightmare

Why Inverse Kinematics (IK) Exists

The limitations of motion replay clearly show why IK is necessary:

Problems with replay:

  • Cannot handle object position changes
  • Requires exhaustive recording for each scenario

How IK solves these problems:

  • Calculates joint angles dynamically based on target position
  • Adapts to any object location in real-time
  • One algorithm covers infinite possible positions

Object Detection for Robot Vision

We can use object detection and camera projections to calculate where an object is based on an image.

How it works:

  • Camera captures image of the workspace
  • Object detection model identifies objects and bounding boxes
  • Using camera calibration, we project 2D pixel coordinates to 3D world coordinates
  • IK calculates the joint angles to reach the target position

Pre-trained Object Detection Models:

  • YOLOv8 - Fast and accurate
  • DETR - Transformer-based detection
  • SSD (Single Shot Detector) - Real-time detection

Can We Simplify the IK Pipeline?

Question: Can we simplify all these middle steps into a single module?

Yes! The Answer is ACT

ACT (Action Chunking Transformer) simplifies the entire pipeline into one end-to-end model!

What is ACT?

ACT (Action Chunking Transformer) is a learning-based approach that:

  • Takes camera images as input
  • Directly outputs robot joint angles
  • Learns from human demonstration data
  • Predicts sequences of actions (action chunks)
  • No need for separate object detection
  • No need for explicit IK calculations
  • One model handles everything from perception to action

Dataset Collection for ACT

ACT learns from human demonstrations - the robot watches and learns!

Ideal Camera Layout

  • Top view - Overall workspace perspective
  • Side/Front view - Depth and approach angle
  • Gripper view - Fine manipulation details

Limitations of ACT 🧠

Limited to One Action Per Model

  • Picking up a cube 🟦 requires one ACT model
  • Picking up a rubber duck 🦆 requires a different ACT model
  • Each new object = new training data + new model

Not Flexible 😓

  • Cannot generalize across different objects
  • Requires retraining for every new task

More ACT 🧠 Limitations

Camera Angle Sensitivity 📷

  • Changing camera angle by just 2.9° causes ACT failure!
  • Model is tightly coupled to camera position

Background Dependency 🖼️

  • Changing the background requires a new ACT model
  • Model learns background-specific features

Method Strengths & Weaknesses

Motion Replay 🔄

  • ✅ Easy to implement
  • ❌ Cannot adapt to changes
  • ❌ Requires recording for every scenario

Inverse Kinematics 🎯

  • ✅ Handles any object position
  • ✅ No training data needed
  • ❌ Requires object detection + calibration

ACT Model 🧠

  • ✅ End-to-end learning
  • ✅ Smooth, natural movements
  • ❌ Needs demonstration data
  • ❌ Sensitive to camera/background changes

Example 1: Mixed Methods (Button Trigger)

Scenario: User presses a button to start a pre-defined sequence

  • 🔘 Button Press triggers the sequence
  • 🧠 ACT picks up the cup (learned behavior)
  • 🧠 ACT picks up fork/spoon (learned behavior)
  • 🎯 IK picks up the cube (precise positioning)

Example 2: Gesture-Controlled Replay

Scenario: User makes hand gestures to trigger different actions

  • 🤚 Hand Gesture detected by camera
  • Gesture recognition routes to appropriate action
  • 🔄 Motion Replay executes pre-recorded sequence for each item

Example 3: VLM-Guided IK

Scenario: User gives text instruction, VLM analyzes scene and decides actions

  • 💬 Text Instruction from user (e.g., “Pick up all items”)
  • 📷 Camera Image captures current scene
  • 👁️ VLM analyzes image and decides which objects to pick
  • 🎯 IK calculates precise positions for each object

Can We Simplify the VLM+IK Pipeline?

Question: Can we simplify the VLM + multiple IK modules into a single module?

Yes! The Answer is VLA

VLA (Vision-Language-Action) model simplifies the entire pipeline into one end-to-end model!

What is VLA?

VLA (Vision-Language-Action) models are large foundation models that:

  • Take camera images and text instructions as input
  • Directly output robot actions (joint angles or end-effector poses)
  • Understand natural language commands
  • Generalize across different objects and environments

Why VLA is not included in this course:

  • 🖥️ High compute requirements - Needs powerful GPUs for inference
  • 📦 Large model size - Models are several GBs, hard to deploy on edge devices
  • 🐌 Slow inference - Not suitable for real-time control without optimization
  • 💰 Expensive training - Requires massive datasets and compute resources
  • 🔧 Complex setup - More engineering overhead compared to simpler methods

For now, focus on mastering 🔄 Motion Replay, 🎯 IK, and 🧠 ACT first!

Custom Gripper Design

You can design your own custom gripper using 3D printing to better suit your specific task!

Why customize your gripper?

  • 🎯 Task-specific design - Create grippers optimized for your particular objects (screwdrivers, bottles, tools, etc.)
  • 🔧 Better grasping - Standard grippers may not work well for irregular shapes
  • 💡 Creative solutions - Design suction cups, soft grippers, or specialized mechanisms
  • 🏭 Rapid prototyping - 3D print and test different designs quickly

Resources:

  • Use CAD software (Fusion 360, Tinkercad, Blender) to design
  • Print with PLA/PETG on standard 3D printers
  • Attach to the SO101 arm’s standard mounting interface

Project Planning: Risk Assessment

When planning your project, be aware of these key risks:

High Risk Components ⚠️

  • 🧠 ACT Model - Requires collecting demonstration data, training time, and may not generalize well
  • 🎯 IK + Object Detection - Camera calibration is tricky, object detection may fail in varying lighting conditions
  • 👁️ VLM Integration - API dependencies, latency issues, and cost concerns

Medium Risk Components ⚡

  • 📷 Camera Setup - Multiple camera synchronization, lighting changes
  • 🔧 Robot Communication - Network latency, connection stability

Low Risk Components ✅

  • 🔄 Motion Replay - Simple to implement, reliable once recorded
  • 💬 Basic UI - Text input, button controls

Key Takeaway: Always have backup plans for high-risk components!

Backup Plan Architecture

Understanding the Backup Plan

Why do we need backup plans?

  • Complex systems have many points of failure
  • Some components may take longer to develop than expected
  • Having fallback options ensures you always have a working demo

Planning Tips:

  • Start developing backup plans early, not as an afterthought
  • Test each backup path independently
  • Know which components are most likely to fail in your setup
  • Communicate backup plans to your team so everyone understands the fallback strategy
